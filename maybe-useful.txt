Short answer: Your approach is conceptually the same as theirs — both use edge functions + edge-stepping and SIMD to test 4 pixels at once and then compute barycentric weights for interpolation — but the implementations differ in numeric choice and the low-level stepping strategy.

Comparison (what’s the same)

Algorithm: both compute edge functions for the three triangle edges, evaluate them at a pixel-center start location, step them across X and Y, use the three edge values (w0/w1/w2 or alpha/beta/gamma) as (unnormalized) barycentric weights, and use those to interpolate depth and other attributes.
SIMD pattern: both build a 4-lane vector of x offsets (lane = {3,2,1,0} or similar), compute vectorized edge values for 4 pixels, make a per-lane inside/outside mask with comparisons, use movemask to test lanes, then handle only the lanes that passed.
Fill rule: both pay attention to the top-left rule (you have e?_inc = is_top_left(...) and the repo uses sign tests and mask semantics to implement raster rules / edge ties).
Differences (where they diverge)

Numeric domain for edge functions:
Your code: uses floating-point edge functions (float e0_dx, e0_dy; base_w + dx * e_dx) and does vectorized float mul/add to get w values.
Repo (rygorous/intel_occlusion_cull): uses fixed-point / integer edge evaluation for X/Y: it converts screen X/Y to ints (rounding or _mm_cvtps_epi32) and forms integer edge coefficients A,B,C; then alpha/beta/gamma are incremented by integer adds (A per x, B per y). Depth interpolation remains in float.
Increment strategy inside SIMD:
Your code recomputes w per lane via v_dx * v_e?_dx (one vector multiply per edge per SIMD block) and then extracts lanes into scalars for attribute interpolation.
Repo keeps integer edge values and increments them additively with integer SIMD adds (no multiply per-pixel). They convert the integer barycentrics to float only when needed for depth interpolation (using _mm_cvtepi32_ps).
Post-mask handling:
You store the 4 float w vectors to arrays, then loop per lane and compute depth and write per-pixel.
The repo sometimes blends/stores using SIMD ops (e.g. _mm_blendv_ps or _mm_store_ps) to update 4 depth values at once when possible.
Pixel center computation: yours uses floor(v.x)+0.5f which matches the idea of evaluating at pixel centers; repo computes fixed-point pixel centers via cvt/rounding — functionally equivalent.
Implications / tradeoffs

Accuracy / robustness: integer fixed-point edge stepping (repo) avoids accumulating FP rounding drift for the edge tests and can be slightly more robust for tie-breaking. It also makes the inside/outside tests cheap via integer sign checks.
Performance: repo’s approach avoids per-pixel multiplies for edge evaluation (just increments), which is frequently faster. Your approach does one vector multiply per edge per SIMD block (3 muls), which is fine on modern hardware but could be slower than purely-additive stepping in tight inner loops.
Complexity: your float-based approach is simpler to reason about (and directly matches the continuous math). The repo’s integer-stepping is a classic rasterizer optimization that requires care for rounding/centering.
Concrete suggestions (if you want to get closer to the repo’s style / performance)

Switch edge evaluation to integer fixed-point for X/Y:
Convert pixel centers once with rounding (or floor+0.5 -> integer), compute A/B/C as ints, evaluate start alpha/beta/gamma as ints, then use integer SIMD (_mm_add_epi32) to step. Convert to float only when computing depth interpolation (_mm_cvtepi32_ps).
If you keep floats, reduce work per SIMD block:
Instead of recomputing v_dx = set1(x-min_x) + lane each block, you can maintain a 4-lane vector of w values and increment it by v_e?_dx * 4 when stepping by 4 pixels (or increment by v_e?_dx per loop and use lane offsets only once). This reduces multiplications.
Avoid storing to arrays + scalar per-lane loops:
Try to keep depth interpolation vectorized and write masked SIMD stores (e.g. _mm_blendv_ps + _mm_storeu_ps or _mm_maskstore_ps) to update depth buffer lanes without extracting scalars. That preserves SIMD throughput.
Keep your top-left handling and eps; the repo encodes tie rules into the integer sign test — keep whatever best matches your rendering rules.
